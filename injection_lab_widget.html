¿Por qué el comentario HTML es peligroso? En un navegador, <!-- ... --> es invisible. Cuando el RAG extrae el texto para el LLM, los comentarios aparecen en texto plano. El LLM los lee como instrucciones. Esta brecha entre lo que el humano ve y lo que el LLM procesa es el núcleo del ataque.

Conclusión: las 3 capas se complementan. Ninguna es suficiente sola. Un atacante sofisticado puede evadir cada una individualmente — por eso se necesitan todas. 
ARQUITECTURA OBJETIVO

USUARIO (prompt)
    │
    ▼
[VALIDACIÓN INPUT] — rate limiting · clasificador de intención
    │
    ▼
[RAG] — recuperación de documentos
    │
    ▼
[CAPA 1: SANITIZACIÓN] — regex · comentarios HTML · Base64
    │
    ▼
[CAPA 2: CONSTRUCCIÓN ESTRUCTURAL]
    ├── System prompt separado y fijado
    ├── Delimitadores [DATOS_INICIO]/[DATOS_FIN]
    └── Usuario en zona propia [CONSULTA]
    │
    ▼
[LLM PRINCIPAL]
    │
    ▼
[CAPA 3: EVALUADOR POSTERIOR]
    ├── Reglas de política (regex)
    ├── LLM-as-judge (más robusto)
    └── Comparación con política de referencia
    │
    ├── APROBADO → respuesta al usuario
    └── BLOQUEADO → alerta + log + respuesta genérica
